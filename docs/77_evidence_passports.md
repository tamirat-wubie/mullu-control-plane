# Evidence Passports

Purpose: define the standard proof-packet read model for governed capability
actions.
Governance scope: [OCE, RAG, CDCV, CQTE, UWMA, SRCA, PRS]
Dependencies: capability passports, gate template registry, evidence passport
schema, and evidence passport validator.
Invariants: every capability has one evidence passport; missing evidence,
approval, blocked action, replay, rollback, and continuation state are explicit;
evidence passports never grant execution authority.

## Architecture

An Evidence Passport answers these questions for each capability:

| Question | Evidence Passport field |
| --- | --- |
| What evidence exists? | `evidence_exists.present_evidence_refs` |
| Who approved it? | `approval.approval_refs` |
| Which approval proof is missing? | `approval.missing_approval_refs` |
| What is missing? | `missing_evidence` |
| What was blocked? | `blocked.blocked_actions` |
| Can it be replayed? | `replay.replayable` |
| Can it be rolled back? | `rollback.can_rollback` and `rollback.can_compensate` |
| Is it safe to continue? | `continuation.safe_to_continue` and `continuation.safe_for_live_action` |

The read model is generated by:

```text
mcoi/mcoi_runtime/app/evidence_passports.py
```

It consumes:

1. capability passports
2. gate template registry

It emits:

1. `examples/evidence_passports.foundation.json`
2. one proof packet per capability
3. summary counters for proof state, outcome, approval, missing evidence,
   replayability, rollback readiness, and continuation safety

## Proof-State Rule

Evidence Passport uses explicit proof states:

| Proof state | Meaning |
| --- | --- |
| `Pass` | The evidence packet is verified enough for its bounded claim. |
| `Fail` | A hard governance block is present. |
| `Unknown` | Required evidence, approval, replay, or rollback proof is missing. |
| `BudgetUnknown` | Resource budget prevented proof closure. |

Foundation projection is conservative:

1. live execution remains disabled;
2. approval is never inferred;
3. missing replay refs remain missing;
4. non-production capabilities remain `AwaitingEvidence`;
5. execution authority stays with governed capability packs, UAO, gates,
   approval policy, receipts, and rollback policy.

For approval-gated capabilities, `approval` also records
`required_approval_gate_ids`, `required_approval_inputs`,
`required_approval_receipts`, `approval_blocked_actions`,
`missing_approval_refs`, and `next_approval_action`. These fields explain what
must be collected, but they do not approve or execute the capability.

## Governance Boundary

These fields must remain fixed:

```text
evidence_passport_set_is_not_execution_authority = true
live_execution_enabled = false
evidence_passport_is_not_execution_authority = true
```

Evidence Passport is a proof packet, not a runtime admission mechanism.
