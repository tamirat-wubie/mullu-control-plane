# 30 -- Organizational Function Runtime

## Purpose

Organizational functions are reusable, policy-bound operational units that
encapsulate repeatable business processes within the Mullu Platform.  Each
function represents a named category of work -- incident response, deployment
review, document intake, approval desk, code review, or any custom type --
with explicit bindings for team assignment, escalation, SLA enforcement,
communication style, review requirements, and outcome tracking.

The function runtime exists to ensure that every recurring operational process
has an explicit contract surface.  No function may operate without a policy
binding, no queue may exist without team assignment, and no SLA may be silently
skipped.

## Owned Artifacts

| Artifact                    | Role                                                        |
|-----------------------------|-------------------------------------------------------------|
| `ServiceFunctionTemplate`   | Identity and metadata for a named organizational function.  |
| `FunctionPolicyBinding`     | Links a function to its policy pack, autonomy mode, deployment profile, and review requirements. |
| `FunctionSlaProfile`        | Target completion time, approval latency target, and escalation threshold for a function. |
| `FunctionQueueProfile`      | Binds a function to a team, default role, escalation chain, communication style, and concurrency limit. |
| `FunctionOutcomeRecord`     | Per-job outcome audit record: completion, escalation, and drift detection. |
| `FunctionMetricsSnapshot`   | Periodic aggregate metrics: job counts, completion times, escalation and drift rates. |

## Function Lifecycle

Every organizational function progresses through a strict lifecycle:

```
draft --> active --> paused --> retired
```

- **draft** -- The function template is defined but not yet bound to queues or
  policies.  No jobs may be routed through a draft function.
- **active** -- The function has all required bindings (policy, SLA, queue) and
  may accept jobs.
- **paused** -- The function is temporarily suspended.  In-flight jobs continue
  but no new jobs are accepted.
- **retired** -- The function is permanently deactivated.  All in-flight jobs
  must complete or be reassigned before retirement finalizes.

Transitions are forward-only except for the `paused <--> active` toggle.  A
retired function cannot be reactivated; a new template must be created.

## Binding Model

Each active function requires the following bindings:

1. **Job queue** -- A `FunctionQueueProfile` that specifies which team handles
   the function's work, the default role for assignment, maximum concurrent
   jobs, and the communication style for notifications.
2. **Team / role assignment** -- The queue profile references a `team_id` and
   `default_role_id`.  These must resolve to existing team and role contracts
   from the team runtime (doc 29).
3. **Escalation chain** -- An optional `escalation_chain_id` within the queue
   profile.  When present, jobs that exceed the SLA escalation threshold are
   routed through this chain.
4. **Communication style** -- One of `formal`, `standard`, `urgent`, or
   `silent`.  Governs the tone and frequency of notifications generated by the
   function's queue.
5. **Review requirements** -- The `FunctionPolicyBinding` declares whether
   human review is required (`review_required`).  When true, no job may
   complete without an explicit approval step.
6. **Retention defaults** -- Inherited from the bound policy pack.  The
   function runtime does not define its own retention; it defers to the policy
   layer.
7. **Deployment profile** -- An optional `deployment_profile_id` in the policy
   binding.  When present, constrains the environments in which the function
   may operate.
8. **Policy pack** -- Every function must reference a `policy_pack_id` through
   its `FunctionPolicyBinding`.  This pack governs autonomy mode, approval
   gates, and audit requirements.

## SLA Model

Each function declares an explicit SLA profile via `FunctionSlaProfile`:

- **target_completion_minutes** -- The expected end-to-end time from job
  creation to completion.  Must be a positive integer.
- **approval_latency_minutes** -- The maximum acceptable time for an approval
  step within the function.  Must be a positive integer.
- **escalation_threshold_minutes** -- The time after which an incomplete job
  triggers escalation.  Must be a positive integer and must not exceed
  `target_completion_minutes`.

No SLA may be silently skipped.  Every job processed through a function
produces outcome records that are evaluated against the SLA profile.

## Outcome Tracking

The function runtime tracks outcomes at two granularities:

### Per-Job Outcomes (`FunctionOutcomeRecord`)

Every job that passes through a function produces an outcome record:

- **completed** -- Whether the job finished successfully.
- **completion_minutes** -- Actual elapsed time from creation to completion.
- **escalated** -- Whether the job triggered escalation.
- **drift_detected** -- Whether the job's execution deviated from the
  function's expected behavior pattern.

### Periodic Metrics (`FunctionMetricsSnapshot`)

Aggregate snapshots are captured periodically and include:

- **total_jobs** -- Jobs entering the function during the period.
- **completed_jobs** -- Jobs that finished successfully.
- **failed_jobs** -- Jobs that ended in failure.
- **avg_completion_minutes** -- Mean completion time for the period.
- **escalation_count** -- Number of jobs that triggered escalation.
- **drift_count** -- Number of jobs with detected drift.

## Prohibitions

1. **No function without explicit policy binding.**  A `ServiceFunctionTemplate`
   in `active` status must have a corresponding `FunctionPolicyBinding`.
2. **No function queue without team assignment.**  A `FunctionQueueProfile`
   must reference a non-empty `team_id` and `default_role_id`.
3. **No silent SLA skip.**  Every job routed through a function must produce a
   `FunctionOutcomeRecord`.  The runtime must evaluate the outcome against the
   function's `FunctionSlaProfile` and flag breaches explicitly.
